## 被 AI 折磨的 15 遍

最开始我让 AI 写一个跳绳计数算法。

**第一遍：** AI 输出一坨代码，我现场架摄像头跳。跳了 20 个，它只数了 12 个。

**第二遍：** 我说"数少了"，AI 改了改。这次数了 28 个——多了一倍。

**第三遍到第十遍：** 在"数少"和"数多"之间反复横跳。我跳了 2000 多个绳，腿都软了。

**第十一遍到第十五遍：** 发现不是计数逻辑的问题，是边界条件——摄像头反光、人站太远、帧率不够——AI 每一遍都在猜我要什么。

**问题核心：AI 在猜，我在跳。猜错一次，我跳一次。这就是 15 遍的来源。**

---

## 顿悟：AI 不是写代码慢，是"对齐需求"太慢

第 15 遍改完后，我坐在地上喘气，突然想明白一件事：

**不是 AI 不会写代码，是 AI 不知道"什么情况下不算"。**

- 我不知道要告诉它"反光的时候别数"
- AI 也不知道要问"如果摄像头丢帧了怎么办"
- 所以我们俩在黑暗里互相摸，摸 15 遍才摸到

那如果我把"必须处理的情况"写成清单，让 AI 一项一项勾选呢？

这就是**八股编码**的起源。

## 什么是八股编码

八股编码是一套**给 AI 看的结构化注释协议**。

不是给人类看的漂亮注释，是给 AI 的"检查清单"。人类觉得"注释多"是特征，不是 bug。

一个函数必须包含这些段落：

```python
# 破题：一句话做什么；一句话不做什么。
# 承题：依赖什么，前置条件是什么。
# [起讲] 设计意图（禁止写实现细节）
# 入手：申请了什么资源（必须成对释放）
def func():
    # ==== 起股 ====
    # 取：获取什么输入
    # 验：验证什么合法性
    
    # ==== 中股 ====
    # 算：核心计算
    # 转：状态转换
    
    # ==== 后股 ====
    # ✓ 正路径：正常流程
    # ✗ 降级路径：异常/降级处理
    
    # ==== 束股 ====
    # 给出：返回什么
    # 留下：副作用/状态变更
```

**关键点：**

1. **破题必须两句**，第一句"做什么"，第二句"不做"。AI 不能再猜范围。
2. **后股必须有降级路径**，哪怕是"N/A"。AI 不能再漏异常处理。
3. **束股必须写副作用**。AI 不能再偷偷改全局状态。
4. **对偶标记**：资源申请 `🔒` 必须对应释放 `🔓`，正路径 `✓` 必须对应降级路径 `✗`。

## 效果：15 遍降到 3-5 遍

用了八股之后，同样的跳绳算法：

**第一遍：** AI 先输出八股框架（只有注释，没有代码）。我检查：
- 破题第二句是不是"不做姿态矫正"？✅
- 起股的"验"里面有没有置信度阈值？✅
- 后股降级路径有没有"摄像头丢帧"？✅

结构对了 = 逻辑对了。我说"通过，填代码"。

**第二遍：** AI 填代码。我跑测试，计数基本准确。

**第三遍：** 微调阈值（TAKEOFF_TH 从 0.45 改到 0.475），±0 精确命中。

**省了什么？**

| 之前（无八股） | 现在（有八股） | 省了 |
|--------------|--------------|------|
| AI写一版→我跳→发现边界漏了→重写 | AI先写框架→我检查"验"里有没有阈值→第一遍钉死边界 | 省掉"猜边界"的迭代 |
| AI写一版→现场崩了→发现没异常处理→重写 | 后股强制降级路径→第一遍就必须写网络中断怎么办 | 省掉"猜异常"的迭代 |
| AI写一版→计数和姿态混在一起→拆不开 | 破题强制"做什么；不做"→第一遍声明"不做姿态矫正" | 省掉"猜范围"的迭代 |
| 我跳15遍累死了→靠体感判断"对了" | 我扫一眼结构标记→判断"结构完整=逻辑完整" | 省掉人肉测试的体力 |

**本质：八股把"事后验收"变成了"事前检查"。** 你不懂代码，但能看懂"有没有降级路径"。

## 双引擎流水线：DeepSeek 预处理 + Kimi 执行

我的完整工作流：

```
我说人话 → DeepSeek 拆成结构化指令 → Kimi Code 按八股输出 → 我检查结构 → 通过 → 填代码
```

**为什么用 DeepSeek 预处理？**
- 便宜。长文本拆解不心疼。
- 强。能把"我想做个跳绳计数"拆成"做什么/不做什么/边界条件/输入输出"。

**为什么用 Kimi 执行？**
- 200K 上下文。装得下我的 4 万行项目 + 八股规范。
- 听话。给八股规范，它就按八股写。

**为什么不用国外模型？**
- GPT-4 1M 上下文贵得离谱，我做项目要跑几百轮，烧不起。
- Claude 也贵，而且它的"思路链"和八股结构不兼容。
- **这不是模型能力问题，是成本控制问题。** 国产组合 + 八股约束 > 单点国外模型。

## 一个对比实验

我找人做了一个滑动窗口限流器的对比：普通编码 vs 八股编码。

| 指标 | 普通编码 | 八股编码 |
|------|---------|---------|
| Bug检索（找异常处理） | ~30秒全文搜索 | ~5秒搜`后股` |
| 资源泄漏检查 | ~60秒人工追踪 | ~15秒对偶核对 |
| 代码审查 | 阅读理解 | 核对清单（快4-6倍）|
| 隐性缺陷 | 4处遗漏 | 全部显性化 |

实际代码只多了 9.8%，93% 的"膨胀"全是结构化注释——不执行，但提供认知导航。

## 你也能用，三步上手

### 第一步：给 Kimi Code 装上八股 Skill

复制我仓库里的 `code_bagu_skill.md`，粘贴到 Kimi Code 的 Skill 设置里。一次配置，永久生效。

### 第二步：项目根目录放 AGENTS.md

复制 `examples/AGENTS_example.md`，改改项目名。告诉 AI："在这个项目里，按八股写。"

### 第三步：每次让 AI 先出框架，不出代码

你的指令：

```
不要写实现。先按八股规范，输出完整框架：
破题、承题、起讲、入手、起股、中股、后股、束股。
我检查结构通过后，你再填充实现。
```

你检查什么：
- 破题第二句是不是"不做"开头？
- 后股有没有 `✗ 降级路径`？
- 束股有没有写"留下"什么副作用？

全部打勾 → "通过，填代码"。就这么简单。
