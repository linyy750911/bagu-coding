## 摘要

本报告通过 3 个真实业务模块的 A/B 对照实验，验证 **Code Bagu（代码八股）约束** 对 AI 编程的量化影响。

**核心结论**：
- 八股约束将 AI 从"语义学哲学家"降维为"结构填空工"
- 自由发挥的 AI 在边界条件上陷入无限递归，甚至导致上下文死机
- 国产模型（DeepSeek v4-pro）+ 八股约束，在生成稳定性上碾压无约束的通用 LLM

### 2.4 Coverage 模式验证（关键修复）

在实验过程中，我们发现旧版 `anti_duality` 规则采用 **strict 1:1 模式**，导致 ✓/✗ 对偶在降级路径多于正路径时频繁误报。

**修复前（strict 模式）预估拦截**：
- 模块1：1✓ + 4✗ → 拦截 3 次
- 模块2：1✓ + 4✗ → 拦截 3 次
- 模块3：1✓ + 5✗ → 拦截 4 次

**修复后（coverage 模式）实际拦截**：
- 规则改为：**✗ ≥ ✓ ≥ 1**（降级路径可多于正路径，但不能没有）
- 3 个模块全部 **0 误报**
- AI 写代码时 **80% 调符号的时间降到 20%**

### 3.2 第二版：思维链全程记录（病理分析）

同一需求重新跑 OpenCode，**全程记录思考链**。

#### 阶段一：语义递归（死循环起点）

OpenCode 在 **"timeout 是否计入连续失败"** 上反复横跳 5 轮：

> *"timeout 算不算失败？…… 网络断开算不算？…… 如果 timeout 不算，那 3 次 timeout 后 DOWN 不触发？…… 但如果算，网络抖动就误报？…… 我还是不算吧…… 不对，需求说'连续 3 次失败'…… 那失败包括 timeout 吗？……"*

**八股版本怎么解决的？** 框架里直接写了：
```python
# ✗ 降级路径1：请求超时 → 返回 STALE 缓存
```
AI **不需要理解"timeout 算不算"**，它只需要填实现。

#### 阶段二：状态机爆炸（自己造概念）

OpenCode 在思维链里**发明了 3 个不存在的需求状态**：

| 发明的状态 | 最终是否采用 | 结果 |
|-----------|-------------|------|
| `UNCHECKED_FAIL` | ❌ 放弃 | 增加语义复杂度，无实际价值 |
| `DOWN_FIRST` | ❌ 放弃 | 和 DOWN 重复，徒增状态 |
| `ERR_CONNECTION` | ❌ 放弃 | 用 error 字段就能表达 |

**八股版本**：状态机只有 4 个（UP / STALE / DOWN / UNCHECKED），`后股` 里 `✓/✗` 直接映射，不多不少。

#### 阶段三：自我否定（能量耗尽）

思维链最后 1/3 全是 **"OK FINAL DECISION" → "NO" → "OK TRULY FINAL DECISION" → "ARGH"** 的循环。

**最终 OpenCode 收敛到与八股框架完全一致的结论**：
- 首次失败无缓存 → `STALE`（空数据），`failures=1`
- 后续失败有缓存 → `STALE`（返回旧数据），`failures+=1`
- 3 次后 → `DOWN`，触发 alert

**但代价是**：
- 思考链长度：**~8000 字**
- 语义纠结轮数：**5 轮**
- 中间废弃状态：**3 个**
- 生成过程稳定性：**出现上下文死机/卡死**

## 四、关键发现

### 发现 1：八股约束防止了 AI 的"思考死机"

在对比实验中，OpenCode（无约束）在处理 `health_checker.py` 的边界条件时，陷入了长达数千字的语义递归，**最终出现上下文死机/卡死**。

同一需求在 Code Bagu 约束下：
- 框架阶段已明确 `✗ 降级路径：请求超时 → 返回 STALE 缓存`
- AI 无需理解"timeout 算不算失败"，直接按框架填空
- **2 轮对话，3.2 秒完成，零拦截**

**结论**：八股约束不仅规范了代码输出，还通过**结构前置消除了 AI 在语义边界上的无限递归**，显著提升了生成过程的稳定性。

### 发现 2：复杂度与拦截率呈"倒 U 型"

| 复杂度 | 拦截率 | 原因 |
|--------|--------|------|
| 简单 | 33% | AI 对 🔒/🔓 边界没摸清，import 块随手乱标 |
| 中等 | 0% | 吃过模块1的亏，模块2 带戒备实现 |
| 复杂 | 0% | 框架先行策略生效，结构先定死，代码只是填空 |

**最复杂的模块反而零拦截**——说明八股约束的边际成本递减。**结构越复杂，先定框架的收益越大**。

### 发现 3：Coverage 模式救了 ✓/✗

如果恢复旧版 strict 1:1：
- 3 个模块将全部被拦截，累计 **10 次误报**
- AI 80% 时间花在调对偶符号，20% 时间写逻辑

改为 coverage 模式（✗ ≥ ✓ ≥ 1）后：
- **10 次误报直接消失**
- 模块1 的 1✓+4✗、模块2 的 1✓+4✗、模块3 的 1✓+5✗ 全部合法通过

## 五、结论

### 5.1 八股编码 = 软件工程里的"三大纪律八项注意"

| 传统开发痛点 | 八股解法 |
|-------------|---------|
| 核心工程师离职 = 项目瘫痪 | 八股模板在，换谁都能续写 |
| 代码审查靠"阅读理解" | 审查靠"核对清单"，初中生都能做 |
| 改 bug 靠直觉定位 | 搜 `==== 后股 ====`，5 秒定位异常 |
| AI 生成靠模型智商 | AI 生成靠系统约束，便宜模型也能用 |

### 5.2 国产模型 + 八股很能打

本实验使用 **DeepSeek v4-pro**（国产模型）+ **Code Bagu 约束**，在生成稳定性上碾压无约束的通用 LLM。

关键配置：
```bash
CODEBAGU_MODEL=deepseek-v4-pro
```

> 注意：v4-pro 默认开启 thinking mode，需要在 `DeepSeekAdapter` 中保留并回传 `reasoning_content`，否则 API 会返回 400 错误。

### 5.3 一句话总结

**八股不是让 AI 写得更"好"，是让 AI 写得更"准"——用确定性的结构，消除不确定性的语义辩论。**
