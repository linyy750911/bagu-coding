## 关键发现

1. **哪个规则报错最多？**
   - 本次测试仅 `anti_duality` 触发 1 次，根因是 🔒/🔓 strict 模式下的误用（把 import 尝试标记为资源申请）。
   - `bagu_paragraphs`、`tuopu_header`、`format_consistency`、`empty_bagu` 均未触发，说明框架设计阶段已规避。

2. **复杂度增加时，拦截率是否线性增长？**
   - **否**。模块1（简单）反而拦截率最高（50%），模块2/3（中等/复杂）均为 0%。
   - 原因：模块1是首个实现，对 🔒/🔓 strict 模式的边界理解有误；后续模块借鉴经验，一次通过。
   - 这表明 **规范熟悉度** 比 **代码复杂度** 对拦截率的影响更大。

3. **重试是否收敛？**
   - 是。模块1重试1次后通过，模块2/3无重试。
   - 收敛速度极快（首轮学习 → 后续零拦截）。

4. **80% 时间是否仍花在调对偶符号？**
   - **否**。本次测试中，对偶符号调整仅占用约 10% 时间（模块1的 1 次 import 注释修正）。
   - 核心原因是 **coverage 模式** 生效：模块1的 `1✓ + 4✗`、模块2的 `1✓ + 4✗`、模块3的 `1✓ + 5✗` 均未触发误报。
   - 若恢复旧版 strict 模式，模块1的 `✓1 ✗8`（含后股注释）将被拦截，耗时将显著增加。

## 优化建议

1. **🔒/🔓 的语义教育**：首个模块容易把 "import 尝试"、"初始化" 等操作误标为 🔒。建议在提示词中增加示例：🔒 仅用于需要显式释放的物理资源（文件句柄、线程、锁、连接池）。

2. **框架先行策略有效**：先输出八股框架让用户确认，再填充实现，可大幅降低写入拦截率（模块2/3 均为 0%）。建议在 codebagu chat 中强制要求首轮只输出框架。

3. **coverage 模式验证成功**：✓/✗ 的 coverage 模式解决了"降级路径天然多于正路径"的问题。建议保持该配置，并观察更多真实场景数据。

4. **监控埋点已就绪**：`SessionMetrics` + `MetricsReporter` 已接入 codebagu chat，后续可通过真实 LLM 会话收集更丰富的数据（工具调用延迟、规则命中热力图等）。
