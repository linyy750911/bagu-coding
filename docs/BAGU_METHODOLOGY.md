# Bagu 八股化开盒方法论

> **目标**：拿到任何古老代码，能够在 1-3 天内完成"开盒→八股化→分析→改进"的完整闭环。

## 一、核心理念

### 1.1 什么是"八股化开盒"

传统代码分析的问题：
- 读代码 = 盲人摸象，看了后面忘了前面
- 画架构图 = 静态快照，一改动就过时
- 写文档 = 一次性劳动，没人维护

八股化开盒的解决：
- **注释即分析**：写八股的过程就是理解代码的过程
- **注释即契约**：破题/承题/起讲/入手 = 输入/依赖/意图/资源
- **注释即索引**：BaguRAG 让 AI 能读懂代码意图
- **注释即改进基线**：有了完整注释，死代码检测、并发分析才有抓手

### 1.2 四阶段闭环

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  阶段1:开盒  │ → │  阶段2:八股  │ → │  阶段3:分析  │ → │  阶段4:改进  │
│  Understanding│    │  Annotating │    │  Analyzing  │    │  Improving  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
     扫描+分级          分层注释          依赖+状态+风险      死代码+重构
```

## 二、阶段1：开盒（Understanding）

### 2.1 项目接入清单

拿到新项目，先做这 5 件事（30分钟）：

| 序号 | 动作 | 命令/工具 | 产出 |
|---|---|---|---|
| 1 | 统计文件规模 | `find . -type f \| wc -l` | 总文件数、语言分布 |
| 2 | 找出超大文件 | `find . -name "*.java" -exec wc -l {} + \| sort -rn \| head -20` | 上帝类清单 |
| 3 | 检查构建系统 | `ls build.gradle pom.xml package.json` | 技术栈确认 |
| 4 | 检查版本控制 | `git log --oneline \| head -10` | 最后修改时间、活跃度 |
| 5 | 运行现有测试 | `./gradlew test` / `npm test` | 基线可用性 |

### 2.2 超大文件识别与分级

```bash
# 找出所有代码文件并按行数排序
find . -type f \( -name "*.java" -o -name "*.kt" -o -name "*.dart" -o -name "*.js" -o -name "*.ts" \) \
  -exec wc -l {} + 2>/dev/null | sort -rn | head -30
```

**分级标准**：

| 级别 | 行数 | 策略 | 时间预算 |
|---|---|---|---|
| S级 | > 2000 | 人工精修，逐方法分析 | 2-4小时/文件 |
| A级 | 1000-2000 | 半自动：scaffold + 人工审核关键方法 | 30分钟/文件 |
| B级 | 300-1000 | 全自动 scaffold | 5分钟/文件 |
| C级 | < 300 | 全自动 scaffold | 1分钟/文件 |

**关键决策**：超大文件（>2000行）**先不拆分**，用分层注释策略解决。拆分的风险 > 收益，因为：
- 状态耦合深（如 GetX Controller、Spring Service）
- 方法间调用关系复杂
- 拆分引入 bug 的概率高

### 2.3 现有注释清理

古老代码的注释通常是噪音，需要清理：

| 注释类型 | 处理方式 | 示例 |
|---|---|---|
| 纯分割线 | 删除 | `//////////////////////////` |
| 过期注释 | 删除 | `// TODO: 2020年修复` |
| 无意义注释 | 删除 | `// getter` / `// setter` |
| 有内容的行注释 | 保留内容，格式统一 | `// 查询用户` |
| Javadoc/DartDoc | 提取为素材，替换为八股 | 提取描述内容 |

**清理脚本模板**（见 `tools/clean_triple_slash.py`）

## 三、阶段2：八股化（Annotating）

### 3.1 八股格式规范

**类级八股**（1套/文件）：

```dart
/**
 * ===================================================================
 * 模块名:        HomeController
 * 所在包:        lib/app/controller
 * 输入输出:      HTTP/WebSocket / 面板机全局状态
 * 最近修改:      2026-05-16 -- 添加八股规范注释
 * ===================================================================
 */

// [推断] 破题: 集中管理面板机主页所有业务逻辑与状态; 不做底层硬件驱动。
// 承题: 依赖 GetX 状态管理、WebSocket 实时通信、HTTP 网络服务。
// 前置条件: 训练杆配置已加载、设备已激活。
// [起讲] 涵盖初始化、配置加载、学生签到、运动发令、成绩接收、页面导航
// 入手: 初始化 GetX 状态、绑定生命周期、建立网络连接
```

**方法级八股**（分层）：

| 层级 | 方法特征 | 注释深度 | 必填字段 |
|---|---|---|---|
| 核心层 | >50行 / 外部高频调用 / 并发相关 | 完整8段 | 破题/承题/起讲/入手/起股/中股/后股/束股 |
| 普通层 | 20-50行 | 简化4段 | 破题/起讲/起股/束股 |
| 极简层 | <20行 / 私有方法 | 1段 | 破题 |

### 3.2 语言适配点

| 语言 | 注释符号 | 类级位置 | 方法级位置 | 特殊处理 |
|---|---|---|---|---|
| Java | `//` / `/** */` | package后 | 方法签名前 | 排除 getter/setter |
| Kotlin | `//` / `/** */` | 类声明前 | 方法签名前 | 注意 `fun` 关键字 |
| Dart | `//` | import后 | 方法签名前 | 排除 `=>` 箭头函数 |
| JavaScript/TS | `//` / `/** */` | 模块顶部 | 函数声明前 | 注意箭头函数 |
| Python | `#` / `"""` | 模块顶部 | 函数声明前 | 注意 `@decorator` |

### 3.3 并发改造标记规范

在八股注释中嵌入并发标记，让后续改造秒定位：

| 标记 | 含义 | 使用场景 | 示例 |
|---|---|---|---|
| `[并发核心]` | 必须改造，否则并发无法运行 | 发枪流程、成绩接收、状态重置 | `[并发核心] 破题: 启动运动测试...` |
| `[并发关联]` | 需要配合调整 | 签到、配置加载、学生查询 | `[并发关联] 破题: 获取训练杆配置...` |
| `[并发无关]` | 不用改 | UI控制、音频、视频、日志 | `[并发无关] 破题: 播放引导语音...` |
| `[状态单值]` | 全局单值，需改为集合/Map | 所有单实例状态变量 | `// [状态单值] pageState 当前为单值...` |
| `[阻塞点]` | 同步调用或阻塞等待 | sync 调用、await等待 | `// [阻塞点] 等待 WebSocket 成绩推送...` |

### 3.4 写八股时必答的5个问题

每个核心方法写八股时，强制回答：

1. **破题**：这方法做什么？（一句话）
2. **承题**：依赖什么前置条件？（什么必须已准备好）
3. **起讲**：输入输出是什么？（参数、返回值、副作用）
4. **入手**：关键步骤有几步？（3-5个主要步骤）
5. **起股/中股/后股/束股**：
   - 每一步调了什么接口？（API契约）
   - 每一步改了什么状态？（状态变更清单）
   - 异常怎么处理？（降级路径）
   - 完成后留下什么？（副作用）

## 四、阶段3：分析（Analyzing）

### 4.1 三维扫描法

八股化完成后，进行三维扫描：

| 维度 | 扫描内容 | 工具 | 产出 |
|---|---|---|---|
| **调用链** | 所有网络调用、跨模块调用 | `grep -rn "http\|\.post\|\.get\|WebSocket"` | 接口依赖图 |
| **状态图** | 所有全局状态变量、修改点 | `grep -rn "\.obs\|Rx\|setState\|="` | 状态变更图 |
| **行为图** | 所有方法入口、调用关系 | AST / 正则匹配 | 方法调用图 |

**扫描脚本模板**：
- 网络调用扫描：`tools/scan_network_calls.py`
- 状态变量扫描：`tools/scan_state_vars.py`

### 4.2 并发分析 checklist

基于三维扫描结果，回答以下问题：

```markdown
## 并发分析 Checklist

### 1. 状态隔离
- [ ] 全局状态是单值还是按任务ID隔离？
- [ ] 同一状态变量是否在多个方法中读写？
- [ ] 状态重置是否会误杀其他批次？

### 2. 调用链隔离
- [ ] 网络调用是否有顺序依赖？
- [ ] WebSocket 消息分发是按任务ID还是广播？
- [ ] 同一接口是否被多个流程复用？

### 3. 资源竞争
- [ ] 线程池配置是否支持多实例？（core/max/queue）
- [ ] 锁的粒度是全局还是按资源？
- [ ] 是否有单线程池限制了并发？

### 4. 阻塞点
- [ ] 同步调用的阻塞时间？
- [ ] await 等待的回调是否有超时？
- [ ] 轮询/polling 的频率和退出条件？
```

### 4.3 死代码检测

基于 import 数据轻量级检测（无需 AST）：

| 类别 | 判定标准 | 处理建议 |
|---|---|---|
| 疑似死代码 | 有八股注释但无人 import | 人工审查，确认是否反射调用 |
| 从未使用 | 无八股且无人 import | 直接归档或删除 |
| 假阳性 | Controller/配置/入口/测试类 | 已排除，无需处理 |

**已知盲区**：
- 同包引用不需要 import（Java/Dart 同包类直接可见）
- Spring 注入 / GetX 依赖注入不通过 import
- 反射/动态加载（Class.forName、SPI）

**workaround**：对方法数 > 0 且无人 import 的文件，人工抽查是否被同包注入引用。

## 五、阶段4：改进（Improving）

### 5.1 改进优先级矩阵

基于八股注释和扫描结果，按以下矩阵排优先级：

| 影响面 \ 修改成本 | 低成本（<1天） | 高成本（>1天） |
|---|---|---|
| **高影响**（阻塞并发/影响核心流程） | 🔴 立即改 | 🟡 排期改 |
| **低影响**（边缘功能/纯代码质量） | 🟢 顺手改 | ⚪ 不改 |

### 5.2 常见改进模式

| 问题模式 | 改进方案 | 八股注释标记 |
|---|---|---|
| 上帝类（>2000行） | 提取 Mixin/Extension/Delegate | `[待拆分]` |
| 全局单值状态 | 改为 `Map<taskId, State>` | `[状态单值]` |
| 同步阻塞调用 | 改为异步 + 回调/轮询 | `[阻塞点]` |
| 单线程池限制 | 改为线程池组 / 按资源隔离 | `[阻塞点]` |
| 重复网络调用 | 加缓存 / 合并请求 | `[可优化]` |
| 异常处理缺失 | 补全降级路径 | `[风险]` |

### 5.3 改进验证

每次改进后必须验证：

1. **代码合规**：`codebagu check <文件>` 通过
2. **功能回归**：运行现有测试用例
3. **注释同步**：改代码必改注释，保持八股与代码一致
4. **RAG 索引更新**：重新生成 `bagu_rag_index.json`

## 六、工具链

### 6.1 核心工具

| 工具 | 用途 | 输入 | 输出 |
|---|---|---|---|
| `bagu_scaffold.py` | 自动插入八股骨架 | 源代码文件 | 带八股注释的源文件 |
| `codebagu check` | 检查八股合规性 | 源代码文件 | 违规报告 |
| `bagu_map_lite.py` | 生成文件级依赖图 | 项目根目录 | HTML 仪表盘 + AI 诊断报告 |
| `bagu_rag.py` | 语义检索 | 自然语言查询 | 相关代码片段 |
| `bagu_dead_code.py` | 死代码检测 | `bagu_map.json` | 死代码报告 |

### 6.2 工具配置

**codebagu 语言配置**（`src/config/languages.ts`）：
- 按语言配置 `lineComment`、`functionPattern`、`topologyStrategy`
- 置信度前缀：`[人工]` / `[推断]` / `[模板]`
- 权重：`人工×1.0`、`推断×0.7`、`模板×0.3`

**bagu_scaffold 语言适配**：
- Java：`//` 注释，`/** */` 拓扑图头
- Dart：`//` 注释（不用 `///`），`/** */` 拓扑图头
- 排除 `=>` 箭头函数（Dart 单行函数）

| 指标 | 数值 | 经验 |

| 总文件 | 807 | 807 个 Java 文件 |
| 八股化时间 | 2 周 | 从 0% → 100% |

## 八、踩坑记录

### 8.1 工具坑

| 坑 | 原因 | 解决 |
|---|---|---|
| macOS sed 不支持 `\s` | BSD sed 语法差异 | 改用 `[[:space:]]*` |
| macOS sed `-i` 需要备份后缀 | 与 GNU sed 行为不同 | 使用 `-i.bak` + `rm -f *.bak` |
| codebagu 编译后需重启 | `dist/` 缓存 | `npm run build` 后重新执行 |
| bagu_map_lite 正则漏前缀 | 只匹配 `// 破题:`，不匹配 `// [推断] 破题:` | 正则需要加 `(?:\[.*?\]\s+)?` |

### 8.2 流程坑

| 坑 | 原因 | 解决 |
|---|---|---|
| 通用模板清零后 check 失败 | 类级破题缺少承题/起讲/入手 | scaffold 需生成完整类级八股 |
| P3 批量标记后 56 个文件失败 | 类级只有破题，无承题 | 修复脚本自动补全类级段落 |
| BaguRAG 索引从 3191→3439 | 正则修复后多了 248 个带前缀的 | 说明之前正则漏检严重 |
| 死代码 200 个"从未使用" | 同包引用不需要 import | 标记为已知盲区，人工抽查 |

## 九、快速启动模板

拿到新项目，复制以下命令序列：

```bash
# 1. 项目扫描（5分钟）
cd <project-root>
find . -type f \( -name "*.java" -o -name "*.kt" -o -name "*.dart" \) -exec wc -l {} + | sort -rn | head -20 > file_ranking.txt

# 2. 识别超大文件
cat file_ranking.txt | awk '$1 > 2000 {print $2}' > s_level_files.txt
cat file_ranking.txt | awk '$1 > 500 && $1 <= 2000 {print $2}' > a_level_files.txt

# 3. 现有注释清理
python3 tools/clean_comments.py <s_level_files.txt>

# 4. 八股化（分层）
# S级：人工精修
# A级：scaffold + 人工审核
# B/C级：全自动 scaffold

# 5. 合规检查
find . -name "*.dart" -exec codebagu check {} \;

# 6. 扫描分析
python3 tools/scan_network_calls.py
python3 tools/scan_state_vars.py
python3 tools/bagu_dead_code.py

# 7. 生成仪表盘
python3 tools/bagu_map_lite.py .
python3 tools/bagu_rag.py . "你的查询"

# 8. 同步到报告站点
bash tools/sync_bagu_reports.sh
```
